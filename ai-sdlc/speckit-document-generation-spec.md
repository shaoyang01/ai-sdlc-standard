# Speckit Document Generation Specification

## Purpose

This specification defines how AI SDLC Standard generates new Speckit project-private documents and bootstrap reports from a target repository.

Generation must be repeatable, code-driven, and explicit about unknowns. It must not copy concrete facts from legacy Speckit documents or other projects.

Standard-package development may use an existing repository such as `logistics-center` as a semantic gold fixture. In that development context, every governance, workflow, coding, routing, entry-coverage, metadata, and quality-gate rule from the fixture must be represented by the standard package, a project-type profile, or an explicit project override mechanism.

Project bootstrap runtime is different: it does not compare against legacy documents. It generates the target project's new rail directly from this standard package, target repository code, and explicit user-confirmed facts.

## Generated Documents

Project bootstrap may generate:

- `.specify/project-governance-profile.yaml`
- `.specify/entry-coverage-profile.yaml`
- `.specify/business-domain-bootstrap.yaml`
- `.specify/project-context/ProjectCodingGuide.md`
- `.specify/project-context/RepositoryStructure.md`
- `.specify/project-context/ProjectGovernanceOverrides.md`
- `.specify/reports/speckit_generation_report.md`

`specs/**` is not generated by project bootstrap. It is generated later by requirement-level Speckit stages.

`.specify/business_domain/**` is not copied by project bootstrap. It is generated later from target code and user-confirmed business boundaries.

## Target Code Scan

Bootstrap should scan, when present:

| Evidence Type | Examples |
| --- | --- |
| Source roots | `src/main/java`, `src`, `app`, `lib` |
| Modules | directories with `src/main`, build files, package roots |
| Entries | `*Controller`, RPC provider `*Impl`, `*Listener`, `*Consumer`, `*Schedule`, `*Job`, `*Worker`, `*Processor` |
| Service layers | `*Service`, `*ServiceImpl`, `*Manager`, `*DomainService` |
| Persistence | `*Mapper`, `*Dao`, `*DAO`, `*Repository`, mapper XML, SQL files |
| Integrations | RPC clients, HTTP clients, MQ producers and consumers |
| Configuration | YAML, properties, env examples, framework configuration |
| Status and entities | enums, entities, models, DTOs |
| Tests | unit, integration, fixture, and regression tests |

The scan is evidence, not final business interpretation. Domain boundaries still require code evidence plus user confirmation when ambiguous.

## Project Governance Profile

Generate from:

- project path and name
- git remote, when available
- detected language and application type
- source roots and module globs
- standard package source and runtime resolvability
- dual-rail isolation settings
- project-private document paths
- generated product artifact policy

Do not include concrete business domains unless they were already generated from this target repository or confirmed by the user.

## Project Context Documents

### ProjectCodingGuide.md

Use target code to record:

- framework adapters
- common utility classes
- DTO/entity/model conversion patterns
- cache, lock, MQ, RPC, and config conventions
- local implementation exceptions

Detected evidence is a scaffold, not an authoritative project coding rule. A project owner or responsible engineer must review and confirm local rules before new `sdlc-*` Skills treat them as project-level coding standards.

If evidence is insufficient, keep placeholders and record gaps in the generation report.

### RepositoryStructure.md

Use target code to record:

- source roots
- module globs
- module boundaries
- entry locations
- generated or excluded paths

Do not copy module explanations from another project.

### ProjectGovernanceOverrides.md

Default to empty. Add an override only when the project explicitly needs a local exception to shared standard rules and the profile records it under `project_overrides`.

## Entry Coverage Profile

Default Java evidence chain:

```yaml
required_layers:
  - entry
recommended_layers:
  - service
  - manager
  - persistence
allow_missing_layers_with_reason: true
```

This is a recommended default, not a universal Java requirement. Project profiles may tighten or relax layer rules based on code structure.

## Business Domain Bootstrap Profile

Generate configuration for later business-domain generation.

It must say:

- project bootstrap does not include business-domain content
- business domains must be based on target code evidence or user-approved boundaries
- legacy `.specify/memory/**`, `.specify/workflow/**`, and `.specify/coding_guide/**` files are preserved but not read as runtime bootstrap sources

## Reports

### Generation Report

Record:

- code evidence used
- user-confirmed facts
- facts not inferred
- generated files and candidate files
- whether legacy local documents were present and ignored as runtime sources
- whether code evidence completeness check ran

## File Write Rules

1. Preflight all write targets before writing.
2. Refuse to overwrite generated profile files unless the user explicitly permits profile overwrite.
3. Never silently overwrite `.specify/project-context/**`; write `.candidate` files when the target exists.
4. Never silently overwrite `.specify/reports/**`; write timestamped report files when a report already exists.
5. Never write `.specify/memory/**`, `.specify/workflow/**`, or `.specify/coding_guide/**`.
6. Never create `specs/**` during project bootstrap.
7. Keep `library/` ignored through `/library/` in `.gitignore`.

## Blocking Conditions

Stop before writing when:

- a generated profile already exists and profile overwrite is not explicitly allowed
- a legacy source would be used as primary generated content
- an output path points to legacy `.specify/memory/**`, `.specify/workflow/**`, or `.specify/coding_guide/**`
- code evidence is too thin and no user-confirmed boundary exists for requested concrete business facts
