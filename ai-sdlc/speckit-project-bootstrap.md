# Speckit Project Bootstrap

## Purpose

This guide defines how AI SDLC Standard is applied to a target repository.

The standard package is the source of shared Speckit governance rules, node artifact specifications, templates, generation specifications, and bootstrap scripts. A target repository stores only project-specific profile files, generated project-private documents, generated business-domain documents, runtime reports, and optional local overrides.

This is not a repository-to-repository migration process and not a legacy replacement event. Existing repositories can be used as semantic fixtures while developing this standard package, but their business facts must not be copied into another repository.

Project bootstrap is code-driven. The primary source for generated project content is the target repository code plus explicit user-confirmed project facts. Legacy Speckit documents are preserved for legacy workflows, but they are not runtime comparison sources and are not bootstrap inputs for the new rail.

## Bootstrap Data Cases

Use these cases to decide what the bootstrap process should generate.

| Case | Target state | Bootstrap behavior |
| --- | --- | --- |
| Empty Speckit project | No `.specify` directory exists. | Create `.specify` skeleton, project profile, entry coverage profile, and business-domain bootstrap profile. |
| Existing Speckit project | `.specify/memory`, `.specify/workflow`, `.specify/coding_guide`, or old scripts exist. | Keep legacy files untouched; generate new-rail files from standards and target code only. |
| Business-domain project | `.specify/business_domain/**` is missing or stale. | Generate or refresh business-domain documents from the target repository code and user-approved boundaries. |
| Java/Spring backend | The project uses Java, Spring, Dubbo, MyBatis, MQ, schedule, or equivalent backend layers. | Use the generic Java/Spring profile defaults, then let the project profile override module names, entry types, and evidence chains. |

## Standard Package Owns

Shared rules live in `ai-sdlc-standard` and should be referenced by target repositories.

Shared ownership includes:

- lifecycle and phase rules
- DocFlow and artifact storage rules
- change control and Re-Gate rules
- complexity routing rules
- Speckit document governance rules
- Speckit node artifact specifications
- entry coverage audit model
- AI write-lock and authorization rules
- documentation metadata and revision rules
- generic SDD stage sequence
- generic sync and reconciliation principles
- generic business-domain structure rules
- reusable templates for project profiles, entry coverage profiles, and business-domain bootstrap profiles
- bootstrap scripts that create project-owned configuration files
- generation source model, dual-rail isolation, and document generation specifications

## Target Repository Owns

Repository-specific files stay in the target repository.

Project ownership includes:

- `.specify/project-governance-profile.yaml`
- `.specify/entry-coverage-profile.yaml`
- `.specify/business-domain-bootstrap.yaml`
- `.specify/business_domain/**` generated from the target repository code and user-confirmed boundaries
- `.specify/reports/**` generated by local audits
- optional `.specify/coding_guide/**` project overlays
- optional project architecture, repository structure, integration, and deployment documents declared in `.specify/project-governance-profile.yaml`
- optional `.specify/project-context/ProjectGovernanceOverrides.md`
- codebase-specific module layout, source roots, entry types, class naming rules, evidence-chain rules, domain names, code anchors, and glossary terms

The target repository should not store copies of shared `.specify/memory/**` or `.specify/workflow/**` rules unless the project profile explicitly marks them as local overrides. New `sdlc-*` Skills must not use legacy `.specify/memory/**`, `.specify/workflow/**`, or `.specify/coding_guide/**` as normal workflow inputs.

## Bootstrap Output

The project bootstrap step creates only a governance skeleton and project profile files:

```text
.gitignore
.specify/
├── project-governance-profile.yaml
├── entry-coverage-profile.yaml
├── business-domain-bootstrap.yaml
├── project-context/
│   ├── ProjectCodingGuide.md
│   ├── RepositoryStructure.md
│   └── ProjectGovernanceOverrides.md
└── reports/
    └── speckit_generation_report.md
library/
```

`business_domain/**` is not copied by project bootstrap. It is generated by the separate business-domain bootstrap script from the target repository.

`specs/**` is not created by project bootstrap. It is created later by Speckit feature stages such as `sdlc-speckit-specify`.

`library/` is created by project bootstrap as the human handoff root and is added to `.gitignore` as `/library/`.

`.specify/project-context/**` is created by project bootstrap as the new project-private document root for `sdlc-*` Skills. Legacy mixed documents remain untouched.

If a project-context file already exists, bootstrap writes a `.candidate` file instead of silently overwriting the project-private facts.

Bootstrap must not silently overwrite report history. On the first run it may write the canonical report name; on later runs, if that report already exists, it must write a timestamped report file such as `speckit_generation_report.20260701-103000.md`.

## Bootstrap Script

Use:

```bash
scripts/bootstrap-speckit-project.sh <target-project-path>
```

Optional flags:

```bash
scripts/bootstrap-speckit-project.sh <target-project-path> \
  --project-name "<project-name>" \
  --language java \
  --application-type backend \
  --standard-package "<path-or-git-url-to-ai-sdlc-standard>"
```

The script should:

1. Inspect the target repository path.
2. Detect or accept project name, language, application type, source roots, and module globs.
3. Scan target repository code evidence such as entries, services, persistence files, MQ, schedules, configuration, and tests.
4. Create `.specify/project-governance-profile.yaml`.
5. Create `.specify/entry-coverage-profile.yaml`.
6. Create `.specify/business-domain-bootstrap.yaml`.
7. Create `.specify/project-context/ProjectCodingGuide.md`, or `.candidate` when the target exists.
8. Create `.specify/project-context/RepositoryStructure.md`, or `.candidate` when the target exists.
9. Create `.specify/project-context/ProjectGovernanceOverrides.md`, or `.candidate` when the target exists.
10. Create `.specify/reports/speckit_generation_report.md`.
11. Create `.specify/reports/`.
12. Create `library/`.
13. Add `/library/` to the target repository `.gitignore` if it is not already present.
14. Refuse to overwrite existing generated profile files unless `--force-profiles` is provided.
15. Never silently overwrite project-context files.
16. Never silently overwrite report files; preserve repeated bootstrap runs as timestamped report history.
17. Never copy `.specify/business_domain/**` from another repository.

Use `--dry-run` to preview generated content without writing files.

## Project Profile

`project-governance-profile.yaml` tells agents how the standard package applies to this repository.

It should describe:

- project name and repository type
- source roots and module globs
- location of `.specify`, `specs`, and `library`
- standard package version and location
- dual-rail isolation policy
- runtime legacy action, which must be preserve-only and not a new-rail input
- project private documents that must be read for coding, architecture, repository structure, or local governance context
- project-specific overlays and exceptions
- required project-generated documents

Use `templates/project-governance-profile-template.yaml`.

## Project Private Documents

Project private documents are repository-owned documents that are not part of the shared standard package.

Examples:

- `.specify/business_domain/**`
- `.specify/project-context/ProjectCodingGuide.md`
- `.specify/project-context/RepositoryStructure.md`
- `.specify/project-context/ProjectGovernanceOverrides.md`

They must be declared in `.specify/project-governance-profile.yaml` before a Skill treats them as required context.

Rules:

1. Project private documents may add repository facts, codebase constraints, and local implementation details.
2. Project private documents must not redefine shared standard rules unless listed under `project_overrides`.
3. Legacy `.specify/memory/**`, `.specify/workflow/**`, and `.specify/coding_guide/**` are not new-rail content sources.
4. If a project private document conflicts with the standard package, standard package wins unless the conflict is declared as a project override.
5. If a project private document conflicts with `.specify/business_domain/**` on business facts, stop and require project-local reconciliation.
6. If required private documents listed in the profile are missing, stop before the affected Speckit stage.

Legacy mixed documents are preserved for legacy workflows. Project bootstrap must not inventory, compare, copy, or treat them as primary sources for generated new-rail project content.

New `sdlc-*` Skills must not read legacy mixed documents for normal workflow context. They read the split outputs instead:

- shared rules from `${AI_SDLC_STANDARD_HOME}`
- private project facts from `.specify/project-context/**`
- business facts from `.specify/business_domain/**`

If a required private fact exists only in a legacy mixed document, stop and ask for target-code evidence or explicit user confirmation before generating new-rail content.

## Entry Coverage Profile

`entry-coverage-profile.yaml` tells audit tools and agents how to discover code entries and evidence chains in this repository.

It should describe:

- entry types
- path patterns
- class suffixes
- exclude patterns
- service, manager, mapper, DAO, controller, listener, schedule, worker, processor, or equivalent layer rules
- business-chain evidence model
- technical-bridge evidence model
- strict-mode blocking outputs

Use `templates/entry-coverage-profile-template.yaml`.

## Business Domain Bootstrap Profile

`business-domain-bootstrap.yaml` tells agents how to generate `.specify/business_domain/**` from the target repository.

It should describe:

- source roots and code include/exclude patterns
- existing local project documents that may be used as input
- domain discovery strategy
- whether human confirmation is required before creating L1/L2
- business-domain root document expectations
- naming rules for L1/L2/L4
- validation rules

Use `templates/business-domain-bootstrap-template.yaml`.

## Post-Bootstrap Generated Documents

After project bootstrap, a separate business-domain bootstrap step generates repository-owned business-domain documents.

Required root documents:

```text
.specify/business_domain/00BusinessLandscape.md
.specify/business_domain/00UbiquitousLanguage.md
.specify/business_domain/01DomainCatalog.md
```

These files are generated from the target repository. Do not copy them from another repository.

`00BusinessLandscape.md` should include:

- repository business purpose
- fact-source layering
- routing principle
- main business domains
- code anchors for each domain

`00UbiquitousLanguage.md` should include:

- domain terms
- common abbreviations
- entity and status vocabulary
- cross-team wording rules

`01DomainCatalog.md` should include:

- L1/L2 domain index
- main document paths
- planned L4 count or maturity
- routing notes

## Existing `.specify` Handling

When a target repository already has `.specify/**`, treat it as local input, not as the source of standard rules.

For each existing file, classify it as:

| Classification | Meaning | Action |
| --- | --- | --- |
| Shared rule duplicate | Same rule already lives in this standard package. | Do not require it as a local file; preserve existing legacy files untouched. |
| Project profile fact | Module layout, entry types, evidence chain, project paths. | Regenerate from target code or explicit user confirmation into project profile or entry coverage profile. |
| Project business fact | Business domain, glossary, lifecycle, status, code anchor. | Regenerate from target code or explicit user confirmation into business-domain outputs. |
| Project override | Necessary repository-specific exception. | Keep as explicit override in project profile. |
| Runtime report | Audit output under `.specify/reports/**`. | Regenerate locally; do not treat as standard. |
| Obsolete | Invalid or conflicting rule. | Do not bootstrap as active configuration. |

## Existing Project Conflict Policy

Existing project files must not be overwritten silently.

Bootstrap behavior:

- If `.specify/project-governance-profile.yaml`, `.specify/entry-coverage-profile.yaml`, or `.specify/business-domain-bootstrap.yaml` already exists, the bootstrap script stops unless `--force-profiles` is used.
- Use `--dry-run` first for existing projects, then merge project facts manually into the profile files.
- `--force-profiles` is allowed only when the user explicitly accepts replacing the generated profile files.
- Existing `.specify/project-context/**` files are not silently overwritten. Bootstrap writes `.candidate` files unless `--force-context` is explicitly used.
- `library/` may be created when missing.
- `.gitignore` may be created or updated to include `/library/`.
- `.specify/project-context/**` may be created when missing.
- Existing `.specify/memory/**` and `.specify/workflow/**` are not authoritative shared rules. They are legacy local input unless listed under `project_overrides`.
- Existing `.specify/business_domain/**` belongs to the target repository and may be used as local business-domain input. It is not copied from another repository.
- Existing `.specify/reports/**` is runtime output and should be regenerated when audit rules change.

Existing `specs/**` behavior:

- Project bootstrap does not create, delete, or rewrite `specs/**`.
- `specs/**` remains the short-term machine fact source for feature workflows.
- When a Speckit stage targets an existing `specs/{feature}` directory, it must verify that the directory belongs to the same requirement or approved DocFlow source.
- If an existing `specs/{feature}` belongs to a different requirement, stale source, or unknown source, the Skill must stop and request a new feature id, Re-Gate, or explicit user decision.
- Do not rewrite an existing `specs/{feature}/spec.md` merely because the feature name matches.

## Bootstrap Workflow

Use this workflow when applying AI SDLC Standard to a repository.

### Step 1: Read Standard Package

Read from `ai-sdlc-standard`:

- `README.md`
- `ai-sdlc/lifecycle.md`
- `ai-sdlc/artifact-storage.md`
- `ai-sdlc/change-control.md`
- `ai-sdlc/complexity-routing.md`
- `ai-sdlc/standard-package-resolution.md`
- `ai-sdlc/speckit-generation-source-model.md`
- `ai-sdlc/speckit-dual-rail-isolation.md`
- `ai-sdlc/speckit-document-generation-spec.md`
- `ai-sdlc/speckit-document-governance.md`
- `ai-sdlc/speckit-project-bootstrap.md`
- relevant `skills/sdlc-*/` instructions

### Step 2: Inspect Target Repository

Inspect:

- module layout
- source roots
- existing `.specify/**`
- existing `specs/**`
- domain terminology
- major entry points
- existing code architecture and layer names
- project private documents declared in `.specify/project-governance-profile.yaml`, if the profile already exists

### Step 3: Generate Project Profiles

Run the bootstrap script or manually fill templates:

- `.specify/project-governance-profile.yaml`
- `.specify/entry-coverage-profile.yaml`
- `.specify/business-domain-bootstrap.yaml`

Profiles must be explicit enough that another agent can run `sdlc-speckit-pipeline` without reading shared rules from target-local `.specify/memory` or `.specify/workflow` copies.

Profiles must also declare project private documents that affect Speckit execution, including coding guides, repository structure docs, architecture docs, integration docs, deployment notes, and explicit local governance overrides.

### Step 3.1: Generate Project Context Documents

Create when missing, or write `.candidate` files when the target file already exists:

- `.specify/project-context/ProjectCodingGuide.md`
- `.specify/project-context/RepositoryStructure.md`
- `.specify/project-context/ProjectGovernanceOverrides.md`

These files are generated project-private documents defined by `ai-sdlc/speckit-document-generation-spec.md`.

Do not modify legacy `.specify/memory/**`, `.specify/workflow/**`, or existing `.specify/coding_guide/**` files.

New `sdlc-*` Skills must read these generated project-context documents instead of legacy mixed documents.

### Step 4: Generate Business Domain Documents

Run the repository-specific business-domain bootstrap step described by `.specify/business-domain-bootstrap.yaml`:

```bash
scripts/bootstrap-business-domain.sh <target-project-path> --dry-run
scripts/bootstrap-business-domain.sh <target-project-path>
```

The default mode generates pending code-evidence skeletons only.

When `.specify/business-domain-bootstrap.yaml` contains user-confirmed `confirmed_domains`, run confirmed mode:

```bash
scripts/bootstrap-business-domain.sh <target-project-path> --confirmed --dry-run
scripts/bootstrap-business-domain.sh <target-project-path> --confirmed
```

Or pass a domain map explicitly:

```bash
scripts/bootstrap-business-domain.sh <target-project-path> --domain-map .specify/business-domain-bootstrap.yaml --dry-run
```

Confirmed mode may generate routable L1/L2/L4 skeletons:

- `.specify/business_domain/{L1}/{L2}/{L2MainDocument}.md`
- `.specify/business_domain/{L1}/{L2}/{L4Document}.md`
- `.specify/business_domain/{L1}/{L2}/{EntryCoverageDocument}.md`

Do not run confirmed mode unless the domain map was explicitly supplied or confirmed by the user or domain owner. Absence of `confirmed_domains` is a blocking condition, not permission to infer business domains from package names.

Generate:

- `00BusinessLandscape.md`
- `00UbiquitousLanguage.md`
- `01DomainCatalog.md`
- L1/L2/L4 skeletons only from observed business domains or user-approved target domains

### Step 5: Validate Bootstrap

Before using Speckit in the target repository:

- confirm shared rules are referenced from the standard package
- confirm project profiles exist and are readable
- confirm business-domain bootstrap configuration exists
- confirm entry coverage profile can express all important entry types
- confirm business-domain root documents are generated or the bootstrap step is explicitly pending
- confirm project-specific exceptions are explicit

## Pipeline Read Order

When `sdlc-speckit-pipeline` starts:

1. Read standard package rules.
2. Read `.specify/project-governance-profile.yaml`.
3. Read project private documents declared by `.specify/project-governance-profile.yaml` when they are required for the current stage.
4. Read `.specify/entry-coverage-profile.yaml` if entry coverage or sync is needed.
5. Read `.specify/business-domain-bootstrap.yaml` if business-domain documents are missing.
6. If business-domain documents exist, read `.specify/business_domain/00BusinessLandscape.md`.
7. Read `.specify/business_domain/00UbiquitousLanguage.md`.
8. Read `.specify/business_domain/01DomainCatalog.md`.
9. Read only the L1/L2/L4 documents relevant to the current requirement.

Do not read legacy `.specify/memory/**` or `.specify/workflow/**` as authoritative unless the project profile explicitly marks them as project overrides.

## Conflict Resolution

Use this precedence model:

| Conflict | Default decision | How to override |
| --- | --- | --- |
| Standard package vs target-local shared rule copy | Standard package wins. | Declare a project override with reason, owner, and target standard rule. |
| Standard package vs project coding guide | Standard package wins for workflow/gate semantics; project guide wins for codebase-specific implementation details. | Move the local exception into `project_overrides` if it changes shared governance behavior. |
| Standard Java/Spring checklist vs project backend guide | Standard checklist provides generic gate coverage; project guide provides module names, utility classes, local framework adapters, and exceptions. | Declare local implementation rules under `project_private_documents` or `project_overrides`. |
| Legacy mixed document with shared and private sections | New Skills do not read it during normal workflow. | Keep the old document for legacy workflows; generate new project-context files from target code and explicit user confirmation. |
| Project profile vs business-domain document | Business facts come from `.specify/business_domain/**`; routing/config facts come from the profile. | Stop and reconcile when the same business fact conflicts. |
| Legacy `.specify/memory/**` or `.specify/workflow/**` vs standard package | Standard package wins. | Only a profile-declared override may change behavior. |
| Existing `specs/**` vs current DocFlow source | Current feature source must be traceable to the same requirement. | Use Re-Gate, new feature id, or explicit user decision. |

Undeclared conflicts are blocking conditions, not merge prompts.

## Blocking Conditions

Stop bootstrap or pipeline activation when:

- project profile is missing
- entry coverage is required but entry coverage profile is missing
- business-domain documents are missing and bootstrap configuration is missing
- legacy local rules conflict with standard package rules
- project-specific exceptions are implicit or undocumented
- required project private documents are missing
- project private documents conflict with standard rules without an explicit profile override
- project private documents conflict with business-domain facts and the correct source cannot be determined
- the agent would need to guess business domains from package names only

## Non-Goals

This guide does not:

- copy shared standard files into target repositories
- migrate `.specify/business_domain/**` content from another repository
- move project business facts into the standard package
- replace `specs/**` as the short-term feature source
- replace `.specify/business_domain/**` as the long-term repository knowledge source
- implement the full entry coverage audit engine
